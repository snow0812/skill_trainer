# User Twin Studio 产品体验优化留存

## 目的

这份文档用于记录产品交互优化过程中的关键判断、已落地改动、当前仍存在的问题，以及下一步的优化方向。

它不是总设计稿，也不是 PRD，而是产品体验迭代日志。

## 当前产品目标

产品的主目标已经进一步收敛为三条核心流程：

1. 建立分身
2. 校正分身
3. 验证并迭代

对应的真实主链路是：

`Materials -> Understanding -> Training -> Preview -> Publish`

其中最关键的产品闭环是：

`Preview / Run -> Preview / Feedback -> Training / Profile -> Preview / Run`

## 第一阶段：从单页改为阶段式流程

### 发现的问题

- 用户在一个页面内需要完成太多任务
- 系统虽然能做很多事，但用户不知道此刻该先做什么
- 页面之间是功能跳转，不是任务驱动跳转

### 已落地的改动

- 从单页改为六阶段：
  - `Overview`
  - `Materials`
  - `Understanding`
  - `Training`
  - `Preview`
  - `Publish`
- 把复杂内容拆成子页：
  - `Materials / Library`
  - `Understanding / Uncertainty`
  - `Training / Profile`
  - `Training / Claims`
  - `Preview / Run`
  - `Preview / Feedback`
  - `Publish / Exports`

### 结果

- 页面已经不再是“大工作台”
- 主要操作开始拥有更清晰的聚焦页

## 第二阶段：降低首页与顶部动作的认知冲突

### 发现的问题

- 首页同时推荐多个下一步
- 顶部动作、侧边栏、页面 CTA 在同时争夺“下一步”
- 用户能看见所有路，但不知道现在到底该走哪一条

### 已落地的改动

- 首页只保留一个状态驱动主 CTA
- 顶部动作区收敛成：
  - 一个主动作
  - 一个条件性的次动作
- 首页摘要卡减少为更关键的几个维度
- 左侧项目区从“长列表”压缩成：
  - 当前项目
  - 最近项目

### 结果

- 首页更像入口页，而不是控制台
- 用户开始能理解“系统建议我现在做什么”

## 第三阶段：把跳转逻辑从静态导航改成状态驱动

### 发现的问题

- 用户进入产品后，仍然需要自己判断下一步去哪
- 完成一个动作后，不知道下一步该继续哪一页
- 主流程虽然通了，但没有形成明确的完成感

### 已落地的改动

- 根路由会按项目状态自动选择默认落点
- 关键动作完成后会自动跳转到更合理的下一页：
  - 创建项目后 -> `Materials`
  - 上传资料后 -> `Overview`
  - 蒸馏后 -> `Understanding`
  - 保存训练草稿后 -> `Preview / Run`
  - 试运行后 -> `Preview / Feedback`
  - 应用建议后 -> `Training / Profile`
  - 导出后 -> `Publish / Exports`
- 页面中增加轻量的“建议下一步”提示条

### 结果

- 产品开始具备“引导用户走主流程”的能力
- 用户不再完全依赖自己记忆流程顺序

## 当前仍然存在的体验卡点

经过一轮自我审视和流程体验，当前最重要的问题已经从“页面太挤”变成了下面三类：

### 1. 阶段完成感还不够强

虽然系统已经会自动跳转，但用户不一定能明显感知：

- 我刚刚完成了什么
- 为什么现在进入这一页
- 在当前页里应该优先做哪一个动作

### 2. 子任务结构对用户仍然不够显性

虽然阶段内已经有二级子页，但用户主要是进入某阶段后才知道有哪些任务。

这会降低“产品地图”的清晰度。

### 3. `Preview -> Feedback -> Training` 的闭环承接还可以更顺

这是产品最重要的体验环节，但当前仍然更像“若干页面跳转”，还不够像“训练旅程”。

## 当前迭代重点

基于当前阶段，产品体验优化应继续集中在这三项：

1. 增强阶段完成提示
2. 强化左侧当前阶段的子任务可见性
3. 让闭环中的上下文承接更明确

## 本轮准备继续做的事情

### 1. 阶段完成/进入提示

目标：

- 在自动跳转后，明确告诉用户：
  - 刚刚完成了什么
  - 现在为什么来到这里
  - 现在应该先做什么

### 2. 左侧导航展开当前阶段子任务

目标：

- 用户在左侧即可看到当前阶段的二级任务
- 减少“进入阶段后才知道还有子页”的成本

### 3. 压缩页面顶部重复导航

目标：

- 避免左侧导航、顶部子导航、页面 CTA 三者重复表达同一层信息

## 后续方向

如果以上优化完成，接下来最值得继续做的不是再拆页面，而是做“旅程感”增强：

- 阶段进度反馈
- 任务完成后的确认状态
- 来自上一步的上下文提示
- Evidence 能力的心智强化

## 第四阶段：把反馈区从“轻量备注”改成“有效训练入口”

### 发现的问题

`Preview / Feedback` 是训练闭环里最关键的一页，但之前的交互仍然存在三个明显问题：

1. 用户点击反馈标签时会立即提交，导致“选择标签”和“确认提交”混在一起。
2. `补充说明` 输入框更像可选备注，而不像高价值监督信号。
3. 用户不知道自己应该写到什么粒度，容易只留下一句模糊评价，无法支撑高质量训练建议。

### 设计判断

反馈页需要从“快速打标签”升级成“明确训练输入”。这意味着页面应该让用户清楚完成三件事：

1. 先判断这次结果属于哪种偏差
2. 再补一句自己真正会怎么说 / 怎么判断
3. 最后明确提交，让系统基于这条监督信号生成训练建议

### 本轮交互优化原则

- 反馈标签只负责选择，不再承担提交动作
- 输入框标题要从“补充说明”改成更具体的“告诉系统哪里不对”
- 通过动态 placeholder 和示例按钮，降低用户组织反馈语言的成本
- 让用户知道：这句补充会直接影响本轮训练建议质量
- 对负向反馈（`不太像` / `太保守` / `逻辑不对`）提高原因输入权重，避免只打标签不解释

### 预期结果

- 用户更容易理解“先选判断类型，再补原因，再生成建议”的完整路径
- 系统拿到的反馈会更具体，能更稳定地回写到训练建议
- `Preview -> Feedback -> Training` 闭环会更像训练流程，而不是一次轻量吐槽

## 总结

目前产品已经从“功能聚合页”进入“阶段式训练流程”阶段。

接下来的重点不再是继续增加页面，而是让用户在这条主链路上：

1. 更知道自己刚刚做了什么
2. 更知道现在为什么来到这里
3. 更知道下一步应该聚焦什么

只要这三点继续增强，产品会从“好用的工具”进一步靠近“清晰的训练工作流”。
