# User Twin Studio PRD

## 产品定位

User Twin Studio 不是文件管理器，也不是提示词工作台。

它是一款用于训练“用户分身”的本地产品，目标是让用户把自己的真实材料教给系统，生成一版会像自己说话、思考和工作的分身，并通过真实任务持续迭代。

## 核心目标

产品只服务一条主链路：

1. 把自己教给系统
2. 看系统理解得对不对
3. 把不对的地方改成稳定规则
4. 用真实任务验证
5. 固化当前稳定版本

## 北极星体验

用户任何时刻都只需要知道一件事：

`我现在最该做的下一步是什么`

用户不需要理解系统内部的 `claims`、`profile`、`evidence` 等技术概念，产品要把这些复杂度藏在后面，只暴露用户真正需要做的动作。

## 理想用户流程

### 1. 开始

回答的问题：

`这个产品能帮我做什么，我怎么开始？`

这一页只负责：

- 解释产品价值
- 帮用户创建项目
- 告诉用户需要准备什么类型的材料
- 让用户继续上次进度

主动作：

- `开始创建分身`

### 2. 材料

回答的问题：

`系统现在有没有足够材料开始学我？`

这一页只负责：

- 上传材料
- 看当前覆盖度
- 看还缺什么
- 判断是否可以生成第一版

主动作：

- `上传材料`
- 条件满足后切换为 **首次蒸馏**；若已有规则、仅补充资料后刷新理解，则使用 **重新蒸馏**

### 3. 理解（概览）

回答的问题：

`系统现在把我理解成了什么？`

这一页只负责：

- 展示系统当前的一句话摘要
- 展示最关键的 3 个理解维度
- 展示最重要的偏差与不确定项
- 引导用户进入校正

主动作：

- `开始校正这一版`

### 4. 校正

回答的问题：

`我要把哪部分改对？`

这一页只负责：

- 告诉用户本轮最值得改的内容
- 接收从验证带回来的反馈
- 引导用户进入两种聚焦动作：
  - 编辑规则草稿
  - 校正系统判断

校正阶段的目标不是“浏览系统理解”，而是“把偏差改成稳定规则”。

主动作：

- `编辑规则草稿`
- 次动作：`校正系统判断`

### 5. 验证

回答的问题：

`这版分身在真实任务里像不像我？`

这一页只负责：

- 展示当前验证状态
- 引导用户跑真实任务
- 引导用户处理最近一次反馈
- 判断是否可以固化当前版本

主动作：

- `开始验证`

### 6. 固化

回答的问题：

`这版是否值得固化并导出？`

这一页只负责：

- 展示当前版本已具备哪些稳定能力
- 执行导出
- 浏览最近一次导出结果

固化不是平时一直暴露的主工作区，而是验证通过后的结果页。

主动作：

- `导出当前稳定版`

## 信息架构

### 一级导航

- `开始`
- `材料`
- `理解`（进入点为「理解概览」`/summary`，避免「第一版摘要」等易歧义说法）
- `校正`
- `验证`

`固化` 不作为长期并列主阶段，而是作为验证通过后的结果页进入。

### 二级页面

- `/start`
- `/materials`
- `/materials/library`
- `/summary`
- `/summary/current`
- `/summary/uncertainty`
- `/correction`
- `/correction/profile`
- `/correction/claims`
- `/validation`
- `/validation/run`
- `/validation/feedback`
- `/release`
- `/release/exports`

## 页面设计原则

### 1. 阶段首页只做解释、摘要、引导

阶段首页不承担复杂编辑任务。

### 2. 聚焦页只完成一个动作

例如：

- 规则草稿页只编辑规则
- 判断校正页只保留 / 改写 / 删除判断
- 验证页只负责输入任务和看输出
- 反馈页只负责判断偏差并生成训练建议

### 3. 所有跨页动作都要带上下文

每次跳转后都要明确告诉用户：

- 刚刚完成了什么
- 为什么现在来到这里
- 现在应该先做什么

### 4. 训练迭代必须显式成环

产品必须清楚表现这条闭环：

`验证 -> 反馈 -> 加入校正 -> 保存 -> 再验证`

### 5. 导出是结果，不是过程入口

用户不应在尚未验证时频繁看到导出动作。

## 状态机

产品状态统一抽象为：

- `noProject`
- `needsMaterials`
- `readyToGenerate`
- `readyToReview`
- `readyToCorrect`
- `readyToValidate`
- `readyToRelease`

以下逻辑必须由同一个状态源导出：

- 默认落点
- 左侧阶段解锁
- 顶部主动作
- 页面内建议下一步
- 固化页是否显示

## 成功标准

### 用户成功

- 能在 3 分钟内完成第一次项目创建和材料上传
- 能在读完「理解概览」后快速回答“系统现在怎么理解我”
- 能在校正和验证之间形成稳定迭代
- 能明确知道一轮修改是否已经保存并生效

### 产品成功

- 用户不再频繁在多个模块间来回找下一步
- 校正与验证的往返路径清晰
- 导出成为验证后的自然结果，而不是孤立模块
